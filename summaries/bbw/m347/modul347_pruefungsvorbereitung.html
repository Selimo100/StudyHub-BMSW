Container vs. VM Kernunterschiede

MerkmalContainerVirtuelle Maschine
BetriebssystemTeilt Kernel des HostsEigenes vollständiges OS
StartzeitSekunden (sehr schnell)Minuten
RessourcenLeichtgewichtigSchwergewichtig
IsolationProzess-Level (OS-Ebene)Hardware-Level
PortabilitätSehr hochMittel
SicherheitWeniger isoliertStärkere Isolation
BeispieleDocker, PodmanVMware, VirtualBox, Hyper-V
Prüfungs-Merksatz A1G Container virtualisieren auf Betriebssystem-Ebene (teilen den Host-Kernel). VMs virtualisieren auf Hardware-Ebene und brauchen ein eigenes OS.

A1F Anforderungen für containerisierte Dienste

Typische Anforderungen, die für den Einsatz von Containern sprechen:

  • Konsistente Laufzeitumgebungen – „Works on my machine" Problem lösen
  • Schnelle Bereitstellung und Startzeit – Services schnell deployen
  • Einfache Skalierung mehrerer Instanzen – Horizontal skalieren
  • Microservice-Architektur gewünscht
  • CI/CD-Pipeline Integration nötig
Nicht für Container Vollständige Virtualisierung eines Betriebssystems pro Dienst → das ist VMs, nicht Container. Direkter Zugriff auf physische Hardware ohne Isolation → kein Container.

A1E Schwachstellen analysieren & Verbesserungen vorschlagen

Problem: Frontend + Backend gemeinsam in einem Container

  • Schwachstelle 1: Wartbarkeit erschwert – jede Änderung betrifft den gesamten Container
  • Schwachstelle 2: Skalierung eingeschränkt – keine separate Skalierung von Komponenten möglich

Lösung: Container aufteilen → unabhängige Aktualisierung & gezielte Skalierung

B1G Monolith vs. Microservice-Architektur

MerkmalMonolithMicroservices
DeploymentGesamte Anwendung in einem ContainerJeder Service separat deployed
SkalierungNur als Ganzes skalierbarJeder Service separat skalierbar
KomplexitätEinfacher BetriebHöherer Betriebs-/Konfigurationsaufwand
WartungÄnderung betrifft allesUnabhängige Updates möglich
ZusatzdiensteIntegriertAuth-Service, API-Gateway etc. möglich
Prüfungs-Merksatz B1G Monolith = die gesamte Anwendung läuft in einem einzigen Container. Microservice = jeder Service läuft in einem eigenen Container, kann separat skaliert werden.

B1F / B1E Warum Frontend / Backend / Datenbank trennen?

  • Frontend und Backend können unabhängig voneinander gewartet und aktualisiert werden
  • Backend kann bei hoher Last separat skaliert werden, ohne das Frontend zu vervielfältigen
  • Datenbank bleibt isoliert → erhöht Stabilität und Sicherheit

B2G Bestandteile der Container-Technologie

  • Docker Engine – Laufzeitumgebung für Container
  • Dockerfile – Bauplan/Rezept für ein Container Image
  • Container Image – unveränderliches Paket mit App + Abhängigkeiten
  • Container Registry – Speicher für Images (z. B. Docker Hub, GHCR)
  • Docker Compose – Multi-Container-Konfiguration via YAML
  • ❌ Kein Bestandteil: Hypervisor (gehört zur VM-Technologie!)

B2F Entwicklungs- und Betriebsablauf

  1. Anwendung entwickeln + Dockerfile erstellen
  2. Container Image bauen – docker build -t my-app .
  3. Mit docker compose mehrere Services lokal gemeinsam starten und testen
  4. In Cloud oder Server deployen – produktiver Betrieb
Warum docker compose für lokale Entwicklung? Ermöglicht gemeinsames lokales Starten von Frontend, Backend und Datenbank → Zusammenspiel wird früh getestet. Ohne dies: Probleme wie falsche Ports, fehlende DB-Verbindungen treten erst im Betrieb auf.

B2E Reverse Proxy – Netzwerkarchitektur

🌐 Internet
↓
🔀 Reverse Proxy (Port 80/443 öffentlich)
↓
⚙️ Backend (nur intern)
↓
🗄️ Datenbank (nur intern)

Warum nur der Reverse Proxy öffentlich?

Damit nur notwendige Ports öffentlich erreichbar sind und interne Services geschützt bleiben. Backend und Datenbank sind nicht direkt aus dem Internet erreichbar → erhöhte Sicherheit.

D1G Wichtigste Docker-Befehle

# Image herunterladen docker pull nginx # Container starten (detached, Port-Mapping, Name) docker run -d --name web-demo -p 8080:80 nginx # Laufende Container anzeigen docker ps # Logs anzeigen docker logs web-demo # Container stoppen docker stop web-demo # Container löschen docker rm web-demo
Port-Mapping Merksatz -p HOST:CONTAINER → z. B. -p 8080:80 = Port 8080 vom Host wird auf Port 80 im Container gemappt

D2G docker run erklärt

docker run -d -p 5000:80 --name web nginx
  • -d → Container läuft im Hintergrund (detached) → Terminal bleibt frei
  • -p 5000:80 → Port 5000 (Host) auf Port 80 (Container) gemappt → Anwendung erreichbar
  • --name web → Container erhält den Namen "web" → einfachere Verwaltung

D1F docker-compose.yml – Korrekte Struktur

services: web: image: nginx:alpine container_name: d1f-web ports: - "8080:80" restart: unless-stopped
Häufige Fehler in docker-compose.yml port (falsch) → muss ports heissen (Plural!) | Port muss als String in Quotes: "8080:80" | enviroment → korrekt ist environment | Volume-Pfad muss existieren

D2F Image in Registry publizieren

E1G Sicherheit beim Cloud-Zugriff

  • MFA (Multi-Faktor-Authentifizierung) → erhöht Sicherheit am meisten ✅
  • Langes Passwort allein → nicht ausreichend
  • HTTPS → wichtig, aber allein nicht die stärkste Massnahme

Port 22 (SSH) – sicherer Remote-Zugriff zur Administration

Port 80 (HTTP) – Webdienst über den Browser erreichbar

Storage: Persistent Volume (nicht temporärer Container-Speicher!) für Datenbankdaten verwenden

E1F Skalierungsmöglichkeiten in der Cloud

  • Vertikale Skalierung – Mehr CPU/RAM für bestehende Instanz (größer machen)
  • Horizontale Skalierung – Mehr Instanzen hinzufügen (breiter machen)
  • Auto-Scaling – Automatische Anpassung der Instanzanzahl nach Last
  • ❌ "Löschen von Instanzen" ist keine Skalierungsstrategie

E1E Netzwerk-Sicherheitsmassnahmen

  • Nur der Reverse Proxy sollte öffentliche Ports (80/443) besitzen
  • Backend und Datenbank dürfen keine öffentlichen Ports freigeben
  • Kommunikation zwischen Backend und Datenbank über internes Docker-Netzwerk
  • Datenbank akzeptiert ausschliesslich Verbindungen vom Backend

E2E Cloud-Instanz Metadaten

{ "instance-id": "vm-01", "region": "europe-west6", "public-ip": "34.65.10.20" }

Diese Metadaten werden für folgendes verwendet:

  • instance-id → Identifikation der Instanz
  • region → Standort der VM
  • public-ip → Netzwerkzugriff von aussen

Helfen bei Verwaltung, Monitoring und automatisierter Konfiguration.

F1G Qualitätssicherung von Container Images

  • Image Scanning (z. B. Trivy) → überprüft Images auf Sicherheitslücken und veraltete Pakete
  • Regelmässige Updates der Base Images
  • Minimale Images verwenden (alpine, slim)

F1F Wichtige Sicherheitsindikatoren

  • ✓ A Verwendung aktueller Base Images
  • ✗ B Container läuft als root → Sicherheitsrisiko! (nicht empfohlen)
  • ✓ C Minimale Anzahl offener Ports
  • ✓ D Verwendung von Secrets statt Klartext-Passwörtern

F1E Sicherheitslücken in Dockerfiles erkennen

FROM node:latest # ⚠️ PROBLEM: latest = unsicher, nicht reproduzierbar WORKDIR /app COPY . . RUN npm install # ⚠️ PROBLEM: ohne package-lock = keine Kontrolle über Abhängigkeiten CMD ["node", "server.js"] # ⚠️ PROBLEM: läuft als root!
3 Sicherheitsprobleme 1. node:latest → unsicher, nicht reproduzierbar → besser: node:20-alpine
2. Container läuft als root → USER-Direktive hinzufügen: USER node
3. npm install ohne Lockfile → keine Kontrolle über Abhängigkeiten → npm ci verwenden

A1G Easy A – Grundlagen Single Choice
Was ist ein zentraler Unterschied zwischen Containern und virtuellen Maschinen?
  • A Container benötigen immer ein eigenes Betriebssystem
  • B Virtuelle Maschinen teilen sich den Kernel des Host-Systems
  • C Container virtualisieren auf Betriebssystem-Ebene ✓
  • D Virtuelle Maschinen starten schneller als Container
Lösung: C Container virtualisieren auf Betriebssystem-Ebene – sie teilen den Kernel des Host-Systems. VMs hingegen virtualisieren auf Hardware-Ebene und brauchen ein eigenes vollständiges Betriebssystem. Container sind deshalb leichter und starten schneller.
A1F Medium A – Grundlagen Multiple Choice
Welche Anforderungen sprechen typischerweise für den Einsatz von Containern bei einem IT-Dienst?
  • A Konsistente Laufzeitumgebungen ✓
  • B Schnelle Bereitstellung und Startzeit ✓
  • C Vollständige Virtualisierung eines Betriebssystems pro Dienst
  • D Einfache Skalierung mehrerer Instanzen ✓
  • E Direkter Zugriff auf physische Hardware ohne Isolation
Lösung: A, B, D Container bieten konsistente Laufzeitumgebungen (kein "works on my machine"), starten sehr schnell, und lassen sich einfach horizontal skalieren. C beschreibt VMs, E beschreibt Bare-Metal – beides keine Container-Merkmale.
A1E Hard A – Grundlagen Offene Frage
Nenne zwei Schwachstellen eines Dienstes, bei dem Frontend und Backend gemeinsam in einem Container betrieben werden, und erkläre kurz, wie diese durch eine Container-Aufteilung verbessert werden können.
Lösung Schwachstelle 1 – Wartbarkeit: Durch die gemeinsame Ausführung wird die Wartbarkeit erschwert, da jede Änderung den gesamten Container betrifft.

Schwachstelle 2 – Skalierung: Die Skalierung ist eingeschränkt, da nicht einzelne Komponenten separat skaliert werden können.

Verbesserung: Durch eine Aufteilung in mehrere Container können Komponenten unabhängig aktualisiert und gezielt skaliert werden.
B1G Easy B1 – Containerkomposition (Architektur) Single Choice
Welche Aussage beschreibt eine monolithische Architektur korrekt?
  • A Frontend, Backend und Datenbank laufen in getrennten Containern
  • B Jeder Service wird unabhängig deployed
  • C Die gesamte Anwendung läuft in einem einzigen Container ✓
  • D Ein Auth-Service wird separat betrieben
Lösung: C Bei einer monolithischen Architektur läuft die gesamte Anwendung in einem einzigen Container (oder Prozess). A, B und D beschreiben Merkmale von Microservice-Architekturen.
B1F Medium B1 – Containerkomposition (Architektur) Offene Frage
Warum ist bei einer Web-Applikation die Trennung in Frontend-, Backend- und Datenbank-Container sinnvoll? Nenne zwei Gründe und erkläre sie kurz.
Lösung Grund 1 – Unabhängige Wartung: Durch die Trennung der Komponenten können Frontend und Backend unabhängig voneinander gewartet und aktualisiert werden.

Grund 2 – Gezielte Skalierung: Das Backend kann bei hoher Last separat skaliert werden, ohne das Frontend zu vervielfältigen. Die Datenbank bleibt isoliert, was Stabilität und Sicherheit erhöht.
B1E Medium B1 – Containerkomposition Multiple Choice
Welche Aussagen treffen auf eine erweiterte Microservice-Architektur zu?
  • A Jeder Service kann separat skaliert werden ✓
  • B Die Architektur ist weniger komplex als ein Monolith
  • C Zusätzliche Services wie ein Auth-Service sind möglich ✓
  • D Der Betriebs- und Konfigurationsaufwand ist höher ✓
  • E Es wird immer nur ein Container verwendet
Lösung: A, C, D Microservices ermöglichen separate Skalierung (A) und zusätzliche Services wie Auth (C). Der Betriebsaufwand ist jedoch höher (D) – das ist ein Nachteil. Die Architektur ist komplexer (nicht einfacher wie B suggeriert), und es werden mehrere Container verwendet (E falsch).
B2G Easy B2 – Bestandteile Container-Technologie Multiple Choice
Welche der folgenden Bestandteile gehören zur Container-Technologie und werden beim Arbeiten mit Docker verwendet?
  • A Docker Engine ✓
  • B Dockerfile ✓
  • C Container Image ✓
  • D Hypervisor
  • E Container Registry ✓
Lösung: A, B, C, E Docker Engine (Laufzeit), Dockerfile (Bauplan), Container Image (Paket) und Container Registry (Speicher) sind alle Bestandteile der Docker/Container-Technologie. Der Hypervisor gehört zur VM-Technologie (z. B. VMware, VirtualBox) – nicht zu Containern.
B2F Hard B2 – Entwicklung containerisierter Dienste Offene Frage
Ein Entwickler erstellt eine containerisierte Webanwendung mit Frontend, Backend und Datenbank. Beschreibe:
1. in welcher Reihenfolge die Anwendung typischerweise entwickelt und betrieben wird
2. warum docker compose für die lokale Entwicklung sinnvoll ist
3. welche Probleme ohne gemeinsames lokales Testen auftreten könnten
Lösung 1. Reihenfolge: Zuerst wird die Anwendung entwickelt und ein Dockerfile erstellt. Danach wird ein Container Image gebaut. Anschliessend werden mehrere Services mit docker compose gemeinsam gestartet und getestet. Danach kann die Anwendung auf einem Server oder in der Cloud betrieben werden.

2. Warum docker compose? docker compose ermöglicht das gemeinsame lokale Starten mehrerer Services wie Frontend, Backend und Datenbank. Dadurch kann das Zusammenspiel früh getestet werden.

3. Probleme ohne lokales Testen: Ohne gemeinsames lokales Testen können Probleme zwischen den Services erst später im Betrieb auffallen, z. B. falsche Ports, Netzwerkprobleme oder fehlende Verbindungen zur Datenbank.
B2E Easy B2 – Netzwerkdiagramm Single Choice

E1G Easy E1 – Sicherer Cloud-Zugriff Single Choice
Welche Massnahme erhöht die Sicherheit beim Zugriff auf einen Cloud-Anbieter am meisten?
  • A Langes Passwort verwenden
  • B Zugriff ohne Login ermöglichen
  • C Multi-Faktor-Authentifizierung (MFA) aktivieren ✓
  • D Zugriff über HTTPS statt HTTP
Lösung: C MFA (Multi-Faktor-Authentifizierung) schützt am stärksten, da selbst bei gestohlenen Zugangsdaten ein zweiter Faktor (z. B. Handy-Code) benötigt wird. Ein langes Passwort allein kann kompromittiert werden. HTTPS schützt die Übertragung, aber nicht den Account selbst.
E1F Medium E1 – Cloud-Instanz konfigurieren Offene Frage
Eine Cloud-Instanz wurde konfiguriert: Name: web-server-01, Ubuntu 24.04, öffentliche IP aktiviert, Firewall: Port 22 offen, Port 80 offen.
Erkläre die Funktion von Port 22 und Port 80 und beschreibe, warum diese Konfiguration für einen Webserver sinnvoll ist.
Lösung Port 22 (SSH): Wird für SSH verwendet und ermöglicht sicheren Remote-Zugriff auf die Cloud-Instanz zur Administration.

Port 80 (HTTP): Wird für HTTP verwendet und ermöglicht den Zugriff auf den Webdienst über den Browser.

Warum sinnvoll: Die öffentliche IP erlaubt den externen Zugriff auf den Server. Die Firewall erlaubt nur die benötigten Ports und erhöht dadurch die Sicherheit.
E1E Medium E1 – Netzwerk-Sicherheitsmassnahmen Offene Frage
Eine containerisierte Anwendung soll gemäss folgenden Vorgaben betrieben werden:
– Nur der Reverse Proxy darf öffentlich erreichbar sein
– Das Backend darf nur intern erreichbar sein
– Die Datenbank darf ausschliesslich mit dem Backend kommunizieren

Beschreibe zwei Sicherheitsmassnahmen, die bei der Netzwerk-Konfiguration umgesetzt werden sollten.
Lösung Massnahme 1: Nur der Reverse Proxy sollte öffentliche Ports (80/443) besitzen. Backend und Datenbank dürfen keine öffentlichen Ports freigeben.

Massnahme 2: Die Kommunikation zwischen Backend und Datenbank sollte über ein internes Docker-Netzwerk erfolgen. Die Datenbank sollte ausschliesslich Verbindungen vom Backend akzeptieren.
E2G Medium E2 – Storage Single Choice
Welche Storage-Variante eignet sich für persistente Daten einer Datenbank?
  • A Temporärer Container-Speicher
  • B Persistent Volume ✓
  • C RAM
  • D Log-Datei
Lösung: B Ein Persistent Volume speichert Daten dauerhaft, auch wenn der Container neu gestartet oder gelöscht wird. Temporärer Container-Speicher (A) geht beim Neustart verloren. RAM (C) ist flüchtig. Log-Dateien (D) sind nicht für Datenbankdaten geeignet.

  • B Horizontale Skalierung ✓

Lösung: A, B, D Vertikale Skalierung = mehr Ressourcen für bestehende Instanz; Horizontale Skalierung = mehr Instanzen; Auto-Scaling = automatische Anpassung. Das Löschen von Instanzen (C) ist keine Skalierungsstrategie.

Übung 3 – Entscheidungsmatrix erstellen Schwierigkeit: ⭐⭐⭐

Eine Schweizer Bundesbehörde sucht einen Cloud-Anbieter für eine containerisierte Anwendung mit sensiblen Bürgerdaten.

Übung 4 – Dockerfile sicher machen Schwierigkeit: ⭐⭐⭐

Verbessere dieses unsichere Dockerfile. Es soll eine Python Flask App (app.py) sicher containerisieren. Benenne mindestens 4 Sicherheitsverbesserungen:

FROM python:latest WORKDIR /app COPY . . RUN pip install -r requirements.txt CMD ["python", "app.py"]

Verbessertes Dockerfile:
FROM python:3.12-slim # ✓ spezifische Version statt latest WORKDIR /app COPY requirements.txt . # ✓ erst requirements, dann Code (Layer caching) RUN pip install --no-cache-dir -r requirements.txt COPY . . RUN useradd -r -s /bin/false appuser # ✓ eigener User, kein root USER appuser EXPOSE 5000 CMD ["python", "app.py"]
4 Verbesserungen:
1. python:3.12-slim statt latest → reproduzierbar, kleiner
2. Eigener User appuser statt root → Sicherheitsrisiko vermieden
3. pip install --no-cache-dir → kleineres Image
4. requirements.txt zuerst kopieren → besseres Layer-Caching
Übung 5 – Netzwerkarchitektur planen Schwierigkeit: ⭐⭐⭐

Plane eine sichere docker-compose.yml für eine Web-App mit:
– NGINX als Reverse Proxy (Port 80 öffentlich)
– Node.js Backend (nur intern, Port 3000)
– PostgreSQL DB (nur intern, Port 5432)

a) Schreibe die docker-compose.yml mit korrekten Netzwerken
b) Erkläre, warum die DB keinen externen Port-Mapping braucht

a) docker-compose.yml:
services: proxy: image: nginx:alpine ports: - "80:80" # Einziger öffentlicher Port! networks: - frontend depends_on: - backend backend: build: ./backend networks: - frontend # Verbindung zu Proxy - backend-db # Verbindung zur DB environment: - DB_HOST=db # KEIN ports: Mapping! Intern erreichbar via Servicename db: image: postgres:15-alpine networks: - backend-db # Nur im internen Netz volumes: - db_data:/var/lib/postgresql/data environment: - POSTGRES_PASSWORD=secret # KEIN ports: Mapping! networks: frontend: backend-db: volumes: db_data:
b) Warum kein externes Port-Mapping für DB?
Die DB ist nur intern über das backend-db Netzwerk erreichbar. Ein externes Port-Mapping würde die DB direkt aus dem Internet zugänglich machen – grosses Sicherheitsrisiko. Das Backend erreicht die DB via Servicename db intern.
Übung 6 – Vollständiges Szenario Schwierigkeit: ⭐⭐⭐⭐

Du wirst als Entwickler engagiert, um die Infrastruktur einer Arztpraxis-Software zu containerisieren. Die Software hat:
– React Frontend (Port 3000)
– Python Backend API (Port 8000)
– PostgreSQL Patientendaten (Port 5432)
– Redis für Session-Cache (Port 6379)

1. Entscheide: Monolith oder Microservices? Begründe.
2. Welche Sicherheitsanforderungen gelten speziell für Patientendaten?
3. Welche Storage-Variante für die DB? Warum?
4. Wie sieht die Netzwerkarchitektur aus (wer kann mit wem kommunizieren)?
5. Welchen Cloud-Anbieter würdest du empfehlen und warum?

Vollständige Musterlösung

1. Microservices – da jede Komponente unabhängig skaliert werden kann (z. B. API bei vielen Anfragen). Ausserdem: separate Sicherheit pro Service möglich (Patientendaten isoliert).

2. Sicherheitsanforderungen (Patientendaten):
– Verschlüsselung der Daten (at rest + in transit)
– Strikte Zugriffskontrollen (nur Backend hat DB-Zugriff)
– Secrets-Management (keine Passwörter in docker-compose.yml)
– Audit Logs (wer greift wann auf Daten zu)
– DSGVO-Konformität (Daten in der EU/Schweiz)

3. Persistent Volume – Patientendaten müssen permanent gespeichert sein. Temporärer Speicher würde alle Daten beim Neustart verlieren!

4. Netzwerkarchitektur:
Internet → Reverse Proxy (öffentlich) → Frontend & Backend (intern)
Backend ↔ PostgreSQL (internes Netz, nur Backend)
Backend ↔ Redis (internes Netz, nur Backend)
Frontend → Backend (über Reverse Proxy)

5. AWS oder Azure – Beide bieten DSGVO-konforme Regionen in der EU, starkes Secrets-Management, Audit Logs und Compliance-Zertifizierungen für Gesundheitsdaten. Für Schweiz: Swiss-Cloud (EXOSCALE) wäre ideal wegen CH-Datenschutzgesetz.
