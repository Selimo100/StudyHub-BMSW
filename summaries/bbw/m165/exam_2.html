1. Redis in 60 Sekunden

Grundidee

Was ist Redis?

Redis ist eine NoSQL-Datenbank, genauer ein Key-Value-Store. Daten werden sehr schnell verarbeitet, weil Redis stark auf Arbeitsspeicher ausgerichtet ist.

Für die Prüfung

Typische Stärken

Sehr schnell, einfaches Datenmodell, gut für Caching, Sessions, Leaderboards, Counters, Echtzeitdaten und schnelle Reads/Writes.

Wichtig

Grenze

Redis ist nicht automatisch ein perfekter Ersatz für relationale SQL-Datenbanken. Man muss Durability, Backup, Replikation und Berechtigungen bewusst konfigurieren.

Merksatz: Redis ist schnell, weil es einfach und speicherbasiert arbeitet. Genau deshalb muss man sich besonders mit Persistenz, Backup, Replikation und Sicherheit beschäftigen.

Begriffe, die du sicher können musst

Begriff Einfache Erklärung Redis-Bezug
Key Eindeutiger Schlüssel, unter dem Daten gespeichert werden. z. B. user:1001
Value Der Wert zum Key; kann String, Hash, Set, List, JSON usw. sein. z. B. Name, Session-Daten, Warenkorb
ACL Access Control List: Regeln, die bestimmen, was ein User darf. z. B. nur lesen, nur bestimmte Keys, gewisse Befehle verboten
RDB Snapshot-Datei von Redis zu einem bestimmten Zeitpunkt. Schnell für Backups und Neustarts
AOF Append Only File: schreibt Operationen fortlaufend mit. Mehr Durability, aber mehr Overhead
Replica Kopie eines Primary-Servers. Für Ausfallsicherheit und mehr Leselast
Sentinel Überwacht Redis und organisiert Failover. Wichtig bei aktiv-passiv / high availability
Cluster / Sharding Verteilt Daten auf mehrere Knoten. Horizontale Skalierung
Projection Nur die benötigten Felder zurückgeben statt alles. z. B. HMGET statt HGETALL

2. D1G – Zugriffsberechtigungen in Redis

D1G

Bei Zugriffsberechtigungen geht es darum, wer auf Redis zugreifen darf und was genau diese Person oder dieses System tun darf. In Redis bedeutet das vor allem: Benutzer, ACL-Regeln, Befehlsrechte, Key-Muster und das Prinzip der minimalen Rechte.

Was du erklären können musst

  • Benutzer: einzelne Konten für Anwendungen oder Personen
  • Rollen: logische Bündelung von Rechten, z. B. Leser, App, Admin
  • Zugriffsrechte: welche Befehle, Keys oder Channels erlaubt sind
  • Least Privilege: immer nur so viele Rechte wie nötig

Warum ist das wichtig?

  • Schutz vor versehentlichem Löschen von Daten
  • Trennung von Admin, App und Analyse-Zugriff
  • Mehr Sicherheit bei mehreren Anwendungen
  • Fehler in einer App betreffen nicht automatisch alle Daten
Element In Redis Beispiel Nutzen
Benutzer Mit ACL definierte Accounts readonly, analyst, appuser Getrennter Zugang pro System
Rolle Fachlich gedachte Rechte-Sammlung Leser, Read-Write, Admin Einfachere Erklärung in der Prüfung
Befehlsrechte Erlaubte/verbotene Kommandos +GET, +HGET, kein SET, kein DEL Schutz vor gefährlichen Aktionen
Key-Muster Zugriff nur auf gewisse Schlüsselbereiche ~orders:*, ~logs:* Isolation von Datenbereichen
Benutzerstatus User aktiv oder deaktiviert on oder off Kontrolle, ob Login erlaubt ist
Prüfungslogik: Benutzer sagt, wer zugreift. Rolle beschreibt fachlich, welche Art von Benutzer das ist. Zugriffsrechte sagen konkret, welche Befehle und Keys erlaubt sind.
Prüfungsfalle: Nicht nur "Passwort" schreiben. Erwartet wird meist: User + Rolle + konkrete Rechte + Key-Bereich.

ACL-Bestandteile, die du sicher lesen können musst

Bestandteil Bedeutung Beispiel
ACL SETUSER User erstellen oder ändern ACL SETUSER readonly ...
readonly Name des Benutzers frei wählbar
on Benutzer ist aktiviert Login erlaubt
>pass123 Passwort setzen hier: pass123
~* Zugriff auf alle Keys oder gezielt ~orders:*
+GET Befehl GET erlauben nur lesen von String-Keys
+HGET Befehl HGET erlauben lesen einzelner Hash-Felder
+DEL Befehl DEL erlauben löschen von Keys

Exakte Prüfungsfrage zu D1G

Originalstil

Ein Entwickler erstellt in Redis den Benutzer readonly mit folgendem Befehl: 

ACL SETUSER readonly on >pass123 ~* +GET

a) Erklären Sie, was die einzelnen Bestandteile des Befehls bedeuten. (1P)

b) Welche Aktionen darf dieser Benutzer ausführen und welche nicht? (1P)

c) Ordnen Sie die Begriffe korrekt zu / verbinde es: (1P)

Begriff Passende Antwort
Benutzer readonly
Rolle Leser (readonly)
Zugriffsrechte nur GET
Musterlösung zur exakten D1G-Prüfungsfrage

a) Bestandteile

  • ACL SETUSER -> Benutzer erstellen oder ändern
  • readonly -> Benutzername
  • on -> Benutzer aktiv
  • >pass123 -> Passwort
  • ~* -> Zugriff auf alle Keys
  • +GET -> nur GET ist erlaubt

b) Erlaubt / nicht erlaubt

  • Erlaubt: Daten lesen mit GET
  • Nicht erlaubt: schreiben mit SET, löschen mit DEL, andere nicht freigegebene Befehle

c) Zuordnung

  • Benutzer -> readonly
  • Rolle -> Leser (readonly)
  • Zugriffsrechte -> nur GET

Zusätzliche Prüfungslogik für ähnliche Aufgaben

  • Wenn nur +GET erlaubt ist, dann ist es ein Lese-User für String-Werte.
  • Wenn zusätzlich +HGET erlaubt ist, kann der User auch Hash-Felder lesen.
  • Wenn statt ~* ein Muster wie ~orders:* steht, darf der User nur auf diesen Key-Bereich zugreifen.
  • Wenn +DEL ergänzt wird, darf er passende Keys auch löschen.

3. E1G – Backup-Konzepte in Redis

E1G

Bei Redis muss man Persistenz und Backup unterscheiden:

  • Persistenz bedeutet: Daten werden so gespeichert, dass sie einen Neustart überleben.
  • Backup bedeutet: Es gibt eine zusätzliche Sicherung, damit man Daten nach Fehlern, Löschen, Defekt oder Fehlkonfiguration wiederherstellen kann.
Konzept Erklärung Redis-Beispiel Wofür gut?
On-demand Snapshot Punktuelle Sicherung zu einem bestimmten Zeitpunkt BGSAVE erzeugt eine RDB-Snapshot-Datei Schnelles manuelles Backup vor Änderungen
Kontinuierliche Sicherung Änderungen laufend protokollieren AOF schreibt Operationen fortlaufend mit Weniger Datenverlust zwischen Snapshots
Cloud Backup Gemanagerter Backup-Prozess beim Anbieter automatische Backups, Aufbewahrungsregeln, Restore-Funktionen Einfacher Betrieb, Offsite-Schutz
Legacy Backup Ältere oder manuelle Sicherungsmethode kopieren von dump.rdb oder AOF-Dateien auf andere Systeme Einfach, aber oft weniger komfortabel
Hybrid Kombination mehrerer Verfahren RDB + AOF zusammen Balance zwischen Performance und Sicherheit
RDB Snapshot

Vorteile

  • Einfaches, kompaktes Backup
  • Schnell für Wiederherstellung
  • Gut als periodische Sicherung

Nachteile

  • Datenverlust seit letztem Snapshot möglich
  • Nur punktuelle Sicherung
AOF

Vorteile

  • Höhere Nachvollziehbarkeit von Änderungen
  • Weniger Datenverlust bei Crashs
  • Geeignet für laufende Sicherung

Nachteile

  • Mehr Speicherbedarf
  • Mehr I/O und potenziell mehr Overhead
Wichtige Prüfungsdifferenz: Replikation ist nicht dasselbe wie Backup. Eine Replica übernimmt Fehler oder versehentlich gelöschte Daten oft ebenfalls. Ein echtes Backup braucht eine getrennte Sicherung, idealerweise mit Versionierung oder zeitlicher Historie.

Typische Prüfungsantwort

Ein sinnvolles Backup-Konzept für Redis kombiniert regelmässige Snapshots mit kontinuierlicher Persistenz oder externen Cloud-Backups. So kann man einerseits schnell wiederherstellen und andererseits den Datenverlust minimieren.

Wann welches Konzept?

Situation Passendes Konzept Begründung
Vor grösserem Update On-demand Snapshot Schneller Sicherungspunkt vor Risiko
Wichtige produktive Daten AOF oder Hybrid Weniger Datenverlust
Managed Service in Cloud Cloud Backups Automatisch, einfacher Restore
Kleine Lernumgebung RDB + manuelles Kopieren Einfach und verständlich

4. F1G – Skalierung und Replikationsarten in Redis

F1G

Hier geht es um zwei Themen, die oft vermischt werden:

  • Skalierung = mehr Leistung / mehr Kapazität
  • Replikation = Kopien von Daten für Verfügbarkeit und Ausfallsicherheit

4.1 Vertikal vs. horizontal skalieren

Art Erklärung Beispiel mit Redis Vor-/Nachteile
Vertikale Skalierung Einen Server stärker machen Mehr RAM / CPU für einen Redis-Node Einfach, aber begrenzt
Horizontale Skalierung Mehrere Server einsetzen Redis Cluster mit Sharding Sehr skalierbar, aber komplexer

4.2 Replikationsarten

Art Was bedeutet das? Redis-Bezug Wozu gut?
Primary and Replica Ein Primary verarbeitet Writes, Replica(s) halten Kopien Standard-Replikation in Redis High Availability, mehr Leselast
Aktiv-Passiv Ein aktiver Knoten, ein passiver wartet auf Failover Primary + Replica + Sentinel Ausfallsicherheit
Multimaster / Active-Active Mehrere Knoten dürfen schreiben Vor allem Redis Enterprise / Active-Active Geo-Verteilung, lokale Schreibzugriffe
Horizontale Skalierung Daten auf mehrere Primaries/Shards verteilen Redis Cluster Mehr Speicher und Durchsatz
Ganz wichtig: Im klassischen Redis Open Source ist die Standardform Primary-Replica. Multimaster ist kein Standard-Setup wie bei normaler Replikation, sondern eine spezielle Active-Active-Lösung in Redis Enterprise/Cloud-Umgebungen.

Primary-Replica einfach erklärt

Client schreibt -> Primary
Replica 1 <- kopiert Daten
Replica 2 <- kopiert Daten

Nutzen:
- Fällt Primary aus, kann eine Replica übernehmen
- Lesende Zugriffe können verteilt werden

Redis Cluster einfach erklärt

Shard A -> Teil der Keys
Shard B -> anderer Teil der Keys
Shard C -> weiterer Teil der Keys

Nutzen:
- Mehr Daten passen ins System
- Last wird aufgeteilt

4.3 Aktiv-Passiv in Redis

Aktiv-passiv bedeutet: Ein Primary bedient den Betrieb. Die Replica ist bereit, zu übernehmen. Sentinel erkennt einen Ausfall und organisiert das Failover, also den Rollenwechsel.

4.4 Multimaster / Active-Active

Hier dürfen mehrere Standorte oder Knoten schreiben. Das ist stark für geografisch verteilte Systeme, aber deutlich komplexer wegen Konfliktlösung und Konsistenzverhalten. Für eine normale Schulantwort reicht meist: mehrere aktive Schreibknoten, gut für globale Verteilung, aber komplexer als Primary-Replica.

4.5 Was bringt horizontale Skalierung konkret?

  • Mehr Datenvolumen, weil Daten auf Shards verteilt werden
  • Mehr Durchsatz, weil Last auf mehrere Knoten verteilt wird
  • Oft bessere Ausfallsicherheit in Kombination mit Replicas
  • Aber: höherer Betriebsaufwand und mehr Komplexität
Starke Prüfungsformulierung: Skalierung erhöht die Leistungsfähigkeit eines Redis-Systems. Replikation erzeugt Datenkopien für Verfügbarkeit und Ausfallsicherheit. Primary-Replica ist die Standardform in Redis, Aktiv-Passiv nutzt diese mit Failover, und horizontale Skalierung verteilt Daten per Cluster auf mehrere Knoten.

5. G1G – Prinzip des Zugriffs in Redis: Queries und Projections

G1G

In Redis greift man meistens nicht mit komplexem SQL auf Tabellen zu, sondern mit Befehlen passend zum Datentyp. Darum ist das Zugriffsprinzip anders als bei SQL.

5.1 Grundidee

Frage In SQL In Redis
Wie hole ich Daten? Mit SELECT-Abfragen Mit Key-basierten Kommandos wie GET, HGET, LRANGE, SMEMBERS
Wie finde ich Datensätze? Über WHERE, JOIN usw. Oft direkt über den Key oder spezielle Index-/Query-Funktionen
Wie hole ich nur gewisse Felder? Projection, z. B. SELECT name z. B. HGET oder HMGET statt HGETALL

5.2 Query in Redis

Eine Query ist allgemein eine Abfrage. In Redis kann das sehr direkt sein:

  • GET session:123 -> einen String-Wert holen
  • HGET user:1001 name -> genau ein Feld aus einem Hash holen
  • HMGET user:1001 name email role -> mehrere gezielte Felder holen
  • ZRANGE leaderboard 0 9 WITHSCORES -> Bereich aus sortierter Menge holen

5.3 Projection in Redis

Eine Projection bedeutet: nur das zurückgeben, was man wirklich braucht.

Schlecht / unnötig breit Besser / gezielt Warum?
HGETALL user:1001 HMGET user:1001 name email Weniger Daten, gezielter Zugriff
KEYS * gezielte Keys oder strukturierte Key-Namen performanter und sicherer
Ganzes JSON laden nur benötigte JSON-Pfade zurückgeben effizienter
Merksatz: Gute Redis-Zugriffe sind zielgerichtet. Hole möglichst direkt den richtigen Key und möglichst nur die Felder, die du wirklich brauchst.

5.4 Typische Redis-Datentypen und Zugriff

Datentyp Wofür geeignet? Beispielzugriff
String Einzelwerte, Tokens, Sessions GET, SET
Hash Objektähnliche Daten wie User HGET, HMGET, HGETALL
List Reihenfolge wichtig, Queues LRANGE
Set Eindeutige Werte SMEMBERS
Sorted Set Rankings, Scores ZRANGE, ZREVRANGE
JSON Dokumentartige Daten gezielte JSON-Pfade und Query Engine

5.5 Redis-spezifische Antwort für die Prüfung

Das Prinzip des Zugriffs in Redis basiert meist auf einem direkten Zugriff über Keys und datentyp-spezifische Befehle. Queries sind also oft einfacher und zielgerichteter als in SQL. Projections bedeuten in Redis, dass nur einzelne Felder oder Teilbereiche abgefragt werden, zum Beispiel mit HMGET statt HGETALL.

Prüfungsfalle: Nicht so tun, als sei Redis einfach „SQL ohne Tabellen“. Redis arbeitet anders: Der Zugriff orientiert sich am Key und am Datentyp.

6. Häufige Prüfungsfallen

Verwechslungen

  • Replikation ≠ Backup
  • Persistenz ≠ Backup
  • Skalierung ≠ Replikation
  • Redis ≠ relationale SQL-Datenbank

Unscharfe Aussagen

  • „Es ist sicher, weil Passwort“ -> zu oberflächlich
  • „Backup ist einfach eine Kopie“ -> zu ungenau
  • „Multimaster ist Standard bei Redis“ -> falsch
  • „Projection ist einfach irgendeine Query“ -> ungenau
Was Prüfer oft mögen: klare Unterscheidungen, passende Redis-Beispiele und kurze Begründungen mit Vorteil/Nachteil.

9. Lösungen zum Prüfungsvorschlag

Lösung zu Aufgabe 1 - D1G

a) Bestandteile

  • ACL SETUSER -> erstellt oder ändert einen Benutzer
  • analyst -> Benutzername
  • on -> Benutzer ist aktiv
  • >secure99 -> Passwort des Benutzers
  • ~orders:* -> Zugriff nur auf Keys, die mit orders: beginnen
  • +GET -> erlaubt GET für String-Werte
  • +HGET -> erlaubt HGET für Hash-Felder

b) Erlaubt / nicht erlaubt

  • Erlaubt: lesen von passenden String-Keys mit GET
  • Erlaubt: lesen einzelner Felder aus Hashes mit HGET
  • Nicht erlaubt: schreiben mit SET, HSET
  • Nicht erlaubt: löschen mit DEL
  • Nicht erlaubt: Zugriff auf andere Key-Bereiche ausser orders:*

c) Zuordnung

  • Benutzer: analyst
  • Rolle: Leser / Analyst
  • Zugriffsrechte: GET und HGET auf orders:*
Lösung zu Aufgabe 2 - D1G 
ACL SETUSER analyst on >secure99 ~logs:* +GET +HGET +DEL

Erklärung: Das Key-Muster wird von ~orders:* auf ~logs:* geändert. Mit +DEL wird zusätzlich das Löschen erlaubt.

Lösung zu Aufgabe 3 - E1G

RDB Snapshot: Redis speichert zu bestimmten Zeitpunkten den kompletten Datenstand in einer Snapshot-Datei. Vorteil: schnell, kompakt und gut für Restore oder periodische Sicherungen.

AOF: Redis schreibt jede Schreiboperation fortlaufend in eine Datei. Beim Neustart können die Befehle wieder abgespielt werden. Vorteil: deutlich aktuellerer Datenstand und weniger Datenverlust.

Lösung zu Aufgabe 4 - E1G
  • Speed: RDB
  • Datenverlust: AOF
  • Speicherbedarf: RDB braucht meist weniger Speicher

Merksatz: RDB = schneller und kompakter. AOF = aktueller und sicherer gegen Datenverlust, aber schwerer und langsamer.

Lösung zu Aufgabe 5 - E1G

Wenn kein Datenverlust erlaubt ist, würde man AOF wählen, weil jede Schreiboperation mitprotokolliert wird. Dadurch ist der Datenstand beim Wiederherstellen viel aktueller als bei periodischen Snapshots. In einer sehr strengen Umgebung wäre zusätzlich eine Kombination aus AOF + externem Backup am besten.

Lösung zu Aufgabe 6 - F1G

Richtig: Mehr Server

Horizontale Skalierung bedeutet, dass man weitere Knoten oder Server hinzufügt, statt nur einen bestehenden Server stärker zu machen.

Lösung zu Aufgabe 7 - F1G

Sharding: Die Daten werden auf mehrere Knoten aufgeteilt. Jeder Knoten hält nur einen Teil der Daten. Ziel: Lastverteilung und mehr Gesamtkapazität.

Replikation: Dieselben Daten werden auf mehrere Knoten kopiert und synchron gehalten. Ziel: Ausfallsicherheit und teilweise Entlastung bei Lesezugriffen.

Kurz: Sharding verteilt Daten, Replikation kopiert Daten.

Lösung zu Aufgabe 8 - F1G

Hochverfügbarkeit bedeutet, dass das System trotz Ausfall einzelner Komponenten weiter erreichbar bleibt. In Redis erreicht man das typischerweise mit Primary + Replica und einem Failover-Mechanismus wie Sentinel. Fällt der Primary aus, kann eine Replica übernehmen.

Lösung zu Aufgabe 9 - G1G

Redis ist primär ein In-Memory Key-Value Store. Die Daten liegen im RAM und werden über eindeutige Keys angesprochen.

Beispiel-Einträge:

username -> "alice99"
score -> "1500"
online -> "true"
Lösung zu Aufgabe 10 - G1G

Einfaches Beispiel mit String:

SET username "alice99"
GET username

Hash-Beispiel:

HSET user:1001 name "Alice" age 23 city "Bern"
HGET user:1001 name
HGETALL user:1001

In Redis greift man direkt über den Key zu. Bei Hashes kommt zusätzlich der Feldname dazu.

Lösung zu Aufgabe 11 - G1G

Eine Projection bedeutet, dass man nur bestimmte Felder holt und nicht den ganzen Datensatz. Das reduziert Datenmenge und Overhead.

Ausgangsdaten:

HSET user:1001 name "Alice" age 23 city "Bern" email alice@mail.com

Projection 1: nur Name

HMGET user:1001 name

Projection 2: nur Name und City

HMGET user:1001 name city

Abgrenzung: HGETALL user:1001 würde alle Felder laden. HMGET lädt nur die benötigten Felder.
