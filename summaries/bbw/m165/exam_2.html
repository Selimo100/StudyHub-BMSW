1. Redis in 60 Sekunden

Grundidee

Was ist Redis?

Redis ist eine NoSQL-Datenbank, genauer ein In-Memory Key-Value Store. Die Daten liegen primär im RAM, deshalb sind Zugriffe extrem schnell.

Typische Stärke

Wofür nutzt man Redis?

Sessions, Caching, Counters, Leaderboards, schnelle App-Daten, Warteschlangen und andere Fälle, in denen Geschwindigkeit sehr wichtig ist.

Prüfungsfokus

Was du sicher können musst

Nicht nur sagen, dass Redis schnell ist, sondern erklären warum - und welche Folgen das für Sicherheit, Backup und Verfügbarkeit hat.

2. D1G - Zugriffsberechtigungen in Redis

D1G

Hier geht es um die Frage: Wer darf was? In Redis werden Zugriffe über Benutzer und ACL-Regeln gesteuert. Diese Regeln bestimmen, welche Befehle erlaubt sind, auf welche Keys zugegriffen werden darf und welche Aktionen verboten sind.

Was du erklären können musst

  • Benutzer: einzelne Konten für Anwendungen oder Personen
  • Rollen: logische Bündel von Rechten, z. B. readonly oder admin
  • Zugriffsrechte: erlaubte Befehle, Keys oder Muster
  • Least Privilege: immer nur die Rechte geben, die wirklich nötig sind

Warum ist das wichtig?

  • verhindert versehentliches Löschen oder Überschreiben
  • trennt Admin, Anwendung und Analyse sauber
  • erhöht Sicherheit bei mehreren Systemen
  • reduziert Schaden, wenn ein Benutzer kompromittiert wird
BegriffBedeutungRedis-BeispielPrüfungslogik
Benutzerein eigenes Kontoreadonly, appuseridentifiziert, wer zugreift
Rollelogische Art von BerechtigungLeser, Schreiber, Adminordnet Benutzer einem Typ zu
Zugriffsrechtekonkrete erlaubte Aktionen+GET, -DEL, ~session:*legt genau fest, was erlaubt ist
ACLAccess Control ListACL SETUSER ...technische Umsetzung der Rechte

Wichtiger Redis-Befehl

ACL SETUSER readonly on >pass123 ~* +GET

So zerlegst du ihn richtig:

  • ACL SETUSER = User erstellen oder ändern
  • readonly = Benutzername
  • on = Benutzer ist aktiv
  • >pass123 = Passwort setzen
  • ~* = Zugriff auf alle Keys
  • +GET = nur GET ist erlaubt
Starke Prüfungsantwort: Zugriffsberechtigungen in Redis sorgen dafür, dass Benutzer oder Anwendungen nur die Befehle und Datenbereiche nutzen können, die sie wirklich brauchen. Das erhöht Sicherheit, verhindert Fehlbedienung und trennt Verantwortlichkeiten sauber.
Prüfungsfalle: Nicht nur schreiben: "Es gibt ein Passwort". Erwartet wird fast immer: Benutzer + Rolle + konkrete Rechte.

3. E1G - Backup-Konzepte in Redis

E1G

Bei Redis musst du sauber unterscheiden zwischen Persistenz, Backup und Replikation.

  • Persistenz = Daten überleben einen Neustart.
  • Backup = zusätzliche Sicherung für Wiederherstellung nach Fehlern, Defekt oder Löschen.
  • Replikation = Datenkopie auf einen anderen Knoten, vor allem für Verfügbarkeit, nicht als Ersatz für Backup.
MechanismusWie funktioniert es?VorteilNachteil
RDB Snapshotspeichert den kompletten Datenstand zu bestimmten Zeitpunktenschnell, kompakt, gut für RestoreDaten zwischen zwei Snapshots können verloren gehen
AOFschreibt jede Schreiboperation in eine Dateiaktueller, weniger Datenverlustmehr Speicherbedarf, mehr Overhead
Cloud BackupProvider erstellt automatische Sicherungeneinfach, oft mit Aufbewahrung und Restoreabhängig vom Dienst und Setup
Legacy Backupmanuelles Kopieren von Backup-Dateieneinfach verständlichweniger komfortabel, fehleranfälliger

RDB in einem Satz

Redis speichert einen kompletten Stand zu einem Zeitpunkt - gut für schnelle Sicherungen, aber nicht perfekt aktuell.

AOF in einem Satz

Redis protokolliert jede Schreiboperation, damit der Zustand beim Neustart möglichst genau wiederhergestellt werden kann.

Ganz wichtig: Replikation ist kein Backup. Wenn ein Benutzer Daten löscht, wird dieser Fehler oft auch auf die Replica kopiert.

4. F1G - Skalierung und Replikation in Redis

F1G

Hier werden oft Begriffe vermischt. Du musst klar sagen können, was Skalierung und was Replikation ist.

BegriffEinfach erklärtRedis-BeispielZiel
Vertikale Skalierungeinen Server stärker machenmehr RAM / CPU für einen Redis-Servermehr Leistung auf einem Knoten
Horizontale Skalierungmehr Knoten hinzufügenRedis Cluster mit Shardingmehr Speicher und mehr Durchsatz
Replikationgleiche Daten auf mehrere Knoten kopierenPrimary + ReplicaAusfallsicherheit und Leseentlastung
ShardingDaten aufteilenShard 1 hat andere Keys als Shard 2Last verteilen

Wichtige Replikationsarten

  • Primary and Replica: Standard in Redis
  • Aktiv-Passiv: ein aktiver Primary, Replica übernimmt bei Ausfall
  • Multimaster / Active-Active: mehrere Schreibknoten, eher Spezialfall

Hochverfügbarkeit

  • System bleibt trotz Ausfall eines Knotens verfügbar
  • Redis erreicht das über Replikation
  • Sentinel erkennt Ausfälle
  • Failover macht eine Replica zum neuen Primary
Sharding vs. Replikation: Sharding verteilt die Daten auf mehrere Knoten. Replikation kopiert dieselben Daten auf mehrere Knoten. Sharding = Lastverteilung. Replikation = Ausfallsicherheit und teilweise mehr Leseleistung.
Prüfungsfalle: Nicht sagen, dass horizontale Skalierung einfach "mehr RAM" bedeutet. Mehr RAM auf einem Server ist vertikale Skalierung.

5. G1G - Zugriff, Queries und Projections

G1G

Der Zugriff in Redis funktioniert anders als in SQL. Redis arbeitet meistens nicht mit Tabellen und WHERE-Filtern, sondern mit direktem Key-Lookup und datentyp-spezifischen Befehlen.

FrageSQLRedis
Wie hole ich Daten?SELECT ...GET, HGET, HMGET, LRANGE
Wie finde ich Datensätze?WHERE, JOINmeistens direkter Key oder definierte Struktur
Wie hole ich nur einzelne Felder?Projection mit SELECT nameHMGET user:1001 name city

Beispiel Key-Value

"username" -> "alice99"
"score"    -> "1500"
"online"   -> "true"

Beispiel Hash

HSET user:1001 name "Alice" age "23" city "Bern" email "alice@mail.ch"
HGET user:1001 name
HMGET user:1001 name city
HGETALL user:1001

Was ist eine Projection?

Man holt nur die Felder, die man braucht, nicht den ganzen Datensatz. Das spart Datenverkehr und ist effizienter.

Warum ist das in Redis wichtig?

Weil gezielte Zugriffe schneller und oft auch sicherer sind. Die Anwendung bekommt dann nicht mehr Daten als nötig.

Merksatz: Gute Redis-Zugriffe sind direkt, zielgerichtet und datentyp-spezifisch. Für die Prüfung ist HMGET statt HGETALL ein sehr gutes Beispiel.

6. Prüfungsvorschlag aus eurem PDF

Diese Fragen solltest du wirklich sicher können. Auch wenn dein Lehrer einzelne Formulierungen anpasst oder austauscht, decken sie genau die geforderten Lernziele ab. Die Fragen stammen aus eurem Prüfungsvorschlag. Die D1G-, E1G-, F1G- und G1G-Aufgaben sind dort jeweils mit 3 Punkten beschrieben.

D1G - 3 Punkte

Frage

Ein Entwickler erstellt in Redis den Benutzer readonly mit folgendem Befehl:

ACL SETUSER readonly on >pass123 ~* +GET

a) Erklären Sie, was die einzelnen Bestandteile des Befehls bedeuten.

b) Welche Aktionen darf dieser Benutzer ausführen und welche nicht?

c) Ordnen Sie korrekt zu: Benutzer / Rolle / Zugriffsrechte.

Musterlösung aufklappen

a) ACL SETUSER = User erstellen oder ändern, readonly = Benutzername, on = aktiv, >pass123 = Passwort, ~* = Zugriff auf alle Keys, +GET = nur GET erlaubt.

b) Er darf Daten lesen mit GET, aber nicht schreiben mit SET und nicht löschen mit DEL.

c) Benutzer = readonly, Rolle = Leser / readonly, Zugriffsrechte = nur GET.

E1G - 3 Punkte

Frage

Beschreiben Sie zwei konkrete Backup-Mechanismen von Redis und erklären Sie jeweils, wie diese funktionieren. Gehen Sie dabei auf den Unterschied zwischen den beiden Methoden ein.

Musterlösung aufklappen

RDB Snapshot: kompletter Datenstand wird zu bestimmten Zeitpunkten gespeichert. Das ist schnell und platzsparend, aber zwischen zwei Snapshots können Daten verloren gehen.

AOF: jede Schreiboperation wird in einer Datei protokolliert. Beim Neustart kann Redis diese Befehle wieder abspielen. Das ist aktueller, aber langsamer und braucht mehr Speicher.

Unterschied: RDB ist effizienter und schneller, AOF ist genauer und aktueller.

F1G - 3 Punkte

Frage

Ein Start-up erwartet stark wachsende Nutzerzahlen und muss seine Redis-Datenbank skalieren.

a) Was bedeutet horizontale Skalierung?

  • a. den vorhandenen Server mit mehr RAM ausstatten
  • b. mehr Server/Knoten zum System hinzufügen
  • c. die Datenbank auf eine schnellere SSD migrieren
  • d. die Anzahl der Benutzer reduzieren

b) Erklären Sie den Unterschied zwischen Sharding und Replikation in Redis.

c) Was versteht man unter Hochverfügbarkeit und wie wird sie in Redis erreicht?

Musterlösung aufklappen

a) Richtig ist b: mehr Server oder Knoten hinzufügen.

b) Sharding verteilt Daten auf mehrere Knoten. Replikation kopiert dieselben Daten auf mehrere Knoten und synchronisiert sie.

c) Hochverfügbarkeit bedeutet, dass das System trotz Ausfall eines Knotens weiterläuft. Redis erreicht dies mit Primary + Replicas und automatischem Failover.

G1G - 3 Punkte

Frage

1. Was ist Redis primär? Zeichne ein kurzes Beispiel dazu mit 3 Einträgen.

2. Wie greift man in Redis auf einen Wert zu?

3. Was versteht man in NoSQL unter einer Projection?

Musterlösung aufklappen

1. Redis ist primär ein In-Memory Key-Value Store. Beispiel: username -> alice99, score -> 1500, online -> true.

2. Über den direkten Key-Lookup. Bei Hashes z. B. mit HGET oder HMGET. Man muss den Key kennen.

3. Eine Projection bedeutet, dass man nur bestimmte Felder eines Eintrags holt, nicht alles.

Prüfungsstrategie: Lerne diese vier Aufgaben nicht nur Wort für Wort, sondern verstehe jeweils die Unterscheidung: Rechte vs. Passwort, RDB vs. AOF, Sharding vs. Replikation, direkter Zugriff vs. Projection. Genau dort werden Fragen oft leicht verändert.

8. Lösungen zu den Zusatzübungen

Lösung zu Übung 1

Risiko 1: Jede Anwendung könnte Daten löschen oder verändern, die sie gar nicht verändern soll. Risiko 2: Wenn ein Zugang kompromittiert wird, ist gleich das ganze System offen. Besser: getrennte Benutzer, z. B. Shop nur für cart:*, Monitoring nur lesend, Admin mit erweiterten Rechten.

Lösung zu Übung 2

Bei reinen Cache-Daten reicht oft ein einfacheres Konzept, weil Daten notfalls neu erzeugt werden können. Bei Session-Daten ist geringerer Datenverlust viel wichtiger, deshalb ist AOF oder ein hybrides Konzept mit Snapshots sinnvoller.

Lösung zu Übung 3

Mehr RAM auf einem Server = vertikale Skalierung. Zusätzliche Replica = Replikation. Redis Cluster mit mehreren Shards = horizontale Skalierung. Sentinel = Hochverfügbarkeit, weil Ausfälle erkannt und Failover organisiert werden.

Lösung zu Übung 4

HMGET user:1001 name role ist besser als HGETALL, weil nur die benötigten Felder übertragen werden. Das spart Datenverkehr und gibt weniger Informationen preis.

Lösung zu Übung 5

Replikation kopiert den aktuellen Zustand auf andere Knoten. Wenn versehentlich Daten gelöscht oder falsche Daten geschrieben werden, wird dieser Fehler oft ebenfalls repliziert. Ein Backup speichert dagegen einen früheren Stand, auf den man später gezielt zurückgehen kann.

Lösung zu Übung 6

Im Aktiv-Passiv-Modell arbeitet normalerweise ein Primary aktiv. Eine Replica hält eine aktuelle Kopie bereit. Fällt der Primary aus, kann die Replica übernehmen. Sentinel überwacht die Instanzen und organisiert dieses Failover.

Lösung zu Übung 7

Mehr RAM auf demselben Server macht nur einen Knoten stärker und ist deshalb vertikale Skalierung. Horizontal bedeutet dagegen, dass man mehrere Knoten oder Server hinzufügt.

Lösung zu Übung 8

Projections sind schneller, weil weniger Daten gelesen und übertragen werden. Sie sind auch sicherer, weil die Anwendung nur die Informationen erhält, die sie wirklich braucht. Das passt zum Prinzip der minimalen Rechte und minimalen Datenweitergabe.
